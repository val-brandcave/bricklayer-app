Widget Library v1

The building blocks the agent composes from

A widget (a type — bar, scatter, map…) is configured with data into a report (a saved object), which renders as a tile on a dashboard or as an MCP app in chat. One component, two entry points. This library documents every type, both of its forms, the two-layer catalog, and the modal that builds them.

How to read this  — This is Voltagent-DS-styled discovery documentation, not the shipping app. Charts are schematic but built from real POC numbers ($6.5B book · 3,935 appraisals). Every tile carries a provenance line — the "analyst with receipts" signature.

Canonical vocabulary · locked Jul 20

Widget → Report → Dashboard → Tile / MCP app

One sentence: a widget (type) is configured with data into a report (object), which appears as a tile on a dashboard or as an MCP app in chat.

The type Widget

The kind of visualization — a reusable DS primitive. ~14 of them. This is what we build.

The object Report

A saved, data-bound instance ("Avg cap rate by vintage"). Listable, permissioned, droppable onto many dashboards.

The collection Dashboard

A saved collection of reports on a grid canvas. (Ed's "board" — the word is retired.)

The two renders Tile / MCP app

Tile = a report on a dashboard. MCP app = the same report in chat, with Save / Edit.

Guardrails · the build principle

How the agent stays trustworthy

01 · Closed set

Composes, never invents

The agent composes from a closed set of typed widgets — it never draws arbitrary UI. Each widget is a tool with a strict schema; the model picks and parameterizes.

02 · Apps SDK pattern

Data separate from context

Structured data goes to the widget separately from the model's context — a full dataset can feed a chart without flooding the model. (OpenAI Apps SDK pattern.)

03 · Receipts

Verified + provenance

Every tile carries ✓ Verified and a provenance line back to the appraisal page. The "analyst with receipts" made literal — every number is traceable.

UI Concepts · Chat × Dashboard

Two chat surfaces over one dashboard model

The whole product is the interaction between the dashboard and the intelligence sitting beside it — as powerful as Power BI without ever feeling like Power BI. One dashboard canvas of tiles; two ways to talk to it: a right-docked co-working assistant that helps on the page you're on, and a full-page chat where the agent builds charts — and whole boards — on the fly.

How to read this  — This is Voltagent-DS-styled discovery documentation, not the shipping app. Each panel is a schematic mock with annotations calling out the behavior. The chat dock is decided: right (§9) — this document shows why and how it behaves; it is no longer an open A/B question. Numbers are real POC figures ($6.5B book · 3,935 appraisals).

The dashboard canvas

The surface everything else assists

The placement rule

Left when it drives · right when it assists

The design community's decision rule (which settles the Cursor side-swap question Val raised): left = collaborative, iterative co-creation where chat is the app; right = task interpretation within an existing workflow.

Chat is the driver → LEFTnot us

Lovable · v0 · Cursor — "describe the outcome, the AI builds everything." The conversation is the primary surface; the canvas is its output.

Chat assists over dashboards → RIGHTBricklayer

Bricklayer's chat is an assistant over existing dashboards, not the driving force. The dashboard is primary; chat interprets tasks within it. So it docks right.

Decided Dock = right (§9, locked Jul 20). Ed's POC docked chat left and used it to drive board-building; we deliberately set that aside. The full-page threaded chat carries the "driver" case as its own destination rather than as a left rail crowding the board.
MCP apps · two directions

The agent asks, or the agent answers

An MCP app is an unopinionated interactive UI element the agent returns inside chat — not text, not a static image. It runs in two directions, both established patterns.

A

Rendered above the composer. The established mechanism is MCP Elicitation / approval cards — a JSON-schema form the client renders. The agent pauses to collect a decision before it acts.

Save-to-dashboard · app-level modal

One save modal — reached from either surface

HubSpot traps the save flow inside the chat drawer; we don't. Save — whether pressed on a dashboard's "Add content" or on an MCP app's Save/Edit in chat — opens the same app-level modal. One editor, two entry points.

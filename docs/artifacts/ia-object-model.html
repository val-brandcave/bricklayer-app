Product Model & IA

The HubSpot-anchored model

Stop inventing an IA. Adopt HubSpot's object model — Dashboards (collections of Reports) plus a Reports library — and layer our real differentiator on top: a genuinely good analytics copilot. HubSpot gives us the familiar, boring, correct BI chassis Ed already has in his head; its own AI ("Breeze") is weak and single-object-bound — precisely the gap we win on.

How to read this  — This is the July 15 course-correction rebase. The July 14 discovery built a bespoke IA (Briefing / Boards / Properties / Deliverables); on the call Cody redirected us onto HubSpot's Reporting skeleton, because that is where the client is drawing his mental model. This document is honest about what changed. Voltagent-DS-styled discovery documentation — not the shipping app. Real POC numbers throughout ($6.5B book · 3,935 appraisals).

The course correction · July 15

What changed

Four honest moves. We retired a bespoke skeleton we were fond of and adopted HubSpot's nouns verbatim — the naming confused the model against the very frame Ed uses. The audit against Ed's 25 POC screenshots confirmed the rebase does not lose him.

The object model · locked vocabulary

Widget → Report → Dashboard → Tile / MCP app

One sentence: a widget (type) is configured with data into a report (object), which appears as a tile on a dashboard — or as an MCP app in chat. Four nouns, adopted to end the term-collision between Ed's, Cody's, and HubSpot's usage.

The type Widget The kind of visualization — bar, line, scatter, map, heatmap, KPI, table… A reusable DS primitive. ~14 of them. This is what we build.
→
The object Report A saved, data-bound instance — "Avg cap rate by vintage." HubSpot's core noun: listable, permissioned, reusable.
→
The collection Dashboard A saved collection of reports on a grid canvas — one purpose or role. (Ed's "board.")
→
Render A Tile The report rendered on a dashboard.
Render B MCP app The same report rendered in chat, with Save / Edit.

Object rules — borrowed from HubSpot so we don't re-litigate them

Reports live standalone

A report can be saved with no dashboard and sit in the library. One report attaches to many dashboards — it is not owned by any one of them.

Access is inherited

Once a report is placed on a dashboard it inherits that dashboard's access. Worth mirroring given our RBAC lenses — permissioning flows through placement.

A default dashboard per lens

Each lens gets a star-set default, changeable via a top-left switcher (All / Favorites / Mine + search) — the HubSpot pattern.

Dashboards · list view

Manage every dashboard in one place

A list / manage view of all dashboards — filterable, with the switcher on the canvas. Deleted dashboards are restorable ~14 days.

Reports · library

User- and AI-made, in one library

Every report — filterable by owner · last-updated · dashboard · tags, with saved views. Crucially, the agent's reports land here too: the agent is just another author.

Why it matters
Because the agent writes into the same library as a human, an AI-built report is a first-class, permissioned, reusable object — not a throwaway chat reply. That is what makes "save this to a dashboard" real.

Placement rule
Chat docks left only when it is the driver of the app (Lovable / v0 / Cursor — "describe the outcome, AI builds everything"). Bricklayer's chat is an assistant over existing dashboards, so we dock it right. Left = collaborative co-creation; right = task interpretation within an existing workflow. Ed's left-dock idea is deliberately set aside.
RBAC · set at provisioning

Three lenses, no switcher

A user's lens is set once, at provisioning. It shapes the data viewpoint — what's surfaced, the default dashboard, how Insights reads. There is no in-app lens switcher: you are your role.

Deliverables · rescoped

Deliverables are actions, not a tab

Ed called export the "average → great" gap. In a Dashboards + Reports world it is not a destination — it's a set of actions hung off dashboards and reports. First-class as a capability, not as a tab.

Positioning · the wedge

CoStar is a database with reports; Bricklayer is an analyst with receipts.

Every number is traceable to the appraisal page. The familiar HubSpot chassis makes us instantly legible; the depth and provenance are what neither CoStar nor HubSpot has. That's the "receipts."

The opening vs HubSpot Breeze

A weak, single-object AI is the gap

HubSpot's Breeze is a prompt-to-single-report generator — single-object only, HubSpot-data-bound, thin iteration. Cody called it "really unimpressive." Our wedge:

  • Cross-object, book-wide reasoning over the whole appraisal data lake — not one-object reports.
  • A real copilot, not a report wizard — co-working + full-page, with write actions and save-to-dashboard first-class.
  • Provenance / receipts — every number traceable to the appraisal page.
